# SURU PKI — Single Root CA

The SURU platform uses **one Root CA**, owned by Tier 4. Every service and client
certificate across all tiers is signed by it. No tier generates a CA of its own.

```text
tier4-operations/pki/certs/root-ca.pem        ← public cert  (safe to share)
tier4-operations/pki/certs/root-ca-key.pem    ← private key  (gitignored, host-only)
```

Generated by [`scripts/generate-certs.sh`](./scripts/generate-certs.sh).

> Both `root-ca.pem` and `root-ca-key.pem` are gitignored in this repository: the
> CA is created locally on first deploy and is specific to your installation. They
> are **never** committed.

---

## Prerequisite order

```text
1. tier4-operations/pki/scripts/generate-certs.sh   # creates Root CA + service certs
2. tier3-core deploy                                 # consumes the Tier 3 service certs
3. tier4-operations deploy                            # generates the frontdoor cert
4. tier1-perimeter: make deploy-full                 # signs + pushes the Tier 1 client cert
```

Tier 1/3/4 steps fail with `SURU Root CA not found` if step 1 has not run.

---

## ⚠️ generate-certs.sh is NOT idempotent for the Root CA

**Every run regenerates `root-ca-key.pem`, silently overwriting the existing
private key.** Only run it when you intend a **full CA rotation**. To add or rotate
a single service cert, use the targeted helpers (e.g.
`tier4-operations/frontdoor/proxy/scripts/generate-frontdoor-cert.sh`) against the
existing CA files.

If the Root CA is rotated, **all** tier certificates must be regenerated and
redeployed in the same change — there is no per-tier rotation without Root CA
rotation.

---

## SAN policy (ingestion backends)

Every ingestion backend server certificate **must** include, as Subject
Alternative Names and in this order:

1. The container's internal Docker DNS name (e.g. `DNS:suru-t3-ingestion-logstash`).
2. Every external SNI hostname routed to it by the Tier 4 frontdoor
   (e.g. `DNS:syslog.suru.local`).
3. `DNS:localhost` — for in-container health probes.

`FRONTDOOR_INGESTION_FQDNS` in `tier4-operations/.env` is the single source of
truth for which SNI hostnames are routed to ingestion. **Every value in it must
appear as a SAN on the corresponding backend cert.** Drift between the two causes
silent ingestion outages (TLS handshake failures the backend logs but the sender
treats as a connection reset).

Verify after any cert change:

```bash
openssl x509 -in tier3-core/certs/logstash.pem -noout -ext subjectAltName
```

## Detecting a CA cert/key mismatch

If `root-ca-key.pem` was overwritten out of sync with `root-ca.pem`, signing fails
with `CA certificate and CA private key do not match`. Confirm with matching
public-key fingerprints:

```bash
openssl x509 -in tier4-operations/pki/certs/root-ca.pem   -pubkey -noout | openssl md5
openssl rsa  -in tier4-operations/pki/certs/root-ca-key.pem -pubout      | openssl md5
# The two digests must be identical.
```

### Recovery (full CA rotation)

```bash
# 1. Regenerate Root CA + Tier 3 service certs
bash tier4-operations/pki/scripts/generate-certs.sh --verbose

# 2. Regenerate the frontdoor cert
bash tier4-operations/frontdoor/proxy/scripts/generate-frontdoor-cert.sh --verbose

# 3. Restart Tier 3 to pick up new service certs
bash tier3-core/scripts/deploy.sh restart --group datalake/opensearch

# 4. Redeploy the Tier 4 frontdoor
bash tier4-operations/scripts/deploy.sh deploy --group frontdoor/proxy

# 5. Regenerate + push the Tier 1 client cert
make -C tier1-perimeter gen-certs deploy
```

---

## Rules

- Never create a new CA with raw `openssl` — always use `generate-certs.sh`.
- Never use self-signed certs that bypass the CA chain.
- Private keys (`*-key.pem`) are gitignored — never commit them; only public certs.
- When adding a new service cert, sign it with `root-ca.pem` / `root-ca-key.pem`,
  mount the `root-ca.pem` alongside it for chain verification, and document it here.
